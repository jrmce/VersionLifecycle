Overview

Version Lifecycle is a multi-tenant SaaS platform designed to help development teams track and manage software deployments across multiple environments.

Whether you're managing microservices, mobile apps, or traditional web applications, Version Lifecycle provides a centralized hub for:

  • Application Management - Organize all your software projects in one place
  • Version Tracking - Follow semantic versioning from draft to production
  • Environment Configuration - Define custom environments (Dev, Staging, Production, etc.)
  • Deployment Monitoring - Track deployment status and history with detailed event timelines
  • Webhook Notifications - Integrate with Slack, Teams, or custom systems

Managing Applications

Applications are the top-level organizational unit in Version Lifecycle. Each application represents a software project you want to track.

üìã What to Include

  • Name: Clear, descriptive name (e.g., "Payment Service", "Mobile App iOS")
  • Description: Brief overview of the application's purpose
  • Repository URL: Link to your source code (GitHub, GitLab, Bitbucket)
  • Application ID: Unique identifier for API integrations

Example: Creating an Application

Name: Customer Portal Description: Public-facing web application for customer self-service Repository: https://github.com/mycompany/customer-portal Application ID: customer-portal-001

Version Tracking

Versions represent specific releases of your application. Follow semantic versioning (MAJOR.MINOR.PATCH) to clearly communicate changes.

Version Statuses

Draft Work in progress, not ready for deployment
Released Ready for deployment to environments
Deprecated Old version, use newer releases
Archived No longer in use, kept for history

Semantic Versioning

MAJOR (1.0.0 ‚Üí 2.0.0)

Breaking changes, incompatible API updates

MINOR (1.0.0 ‚Üí 1.1.0)

New features, backward compatible

PATCH (1.0.0 ‚Üí 1.0.1)

Bug fixes, no new features

Example: Version Progression

v1.0.0 - Initial release (Released) v1.0.1 - Hotfix for login bug (Released) v1.1.0 - Added search feature (Released) v1.0.0 - Now marked as (Deprecated) v2.0.0 - Complete UI redesign (Draft ‚Üí Released)

‚ö° Deployment Workflow

  1. 1. Create Pending Deployment: Select application, version, and target environment
  2. 2. Review Details: Verify configuration before confirming
  3. 3. Confirm Deployment: Transition from Pending to InProgress
  4. 4. Monitor Progress: Watch event timeline for updates
  5. 5. Completion: Status updates to Success/Failed/Cancelled

üõí E-commerce Platform Deployment

Scenario: An online retail company needs to deploy a new checkout feature across multiple environments before reaching production.

Setup:

  • Version: v3.5.0 (new one-click checkout)

iOS App

Workflow:

  3. 3. Deploy to "Beta" environment (TestFlight/Internal Testing)
  4. 4. After successful testing, deploy to "Production" environment
  5. 5. Mark v2.3.x as "Deprecated" once v2.4.0 is in production
